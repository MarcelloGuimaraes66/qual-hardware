# Desenho técnico — Qual Hardware

## Camadas

- Shared: schemas Zod/JSON e tipos versionados, com leitores compatíveis.
- Database: migração v6 exclusivamente aditiva e repositórios de evidência.
- Engine: comparabilidade, cobertura, leave-one-out, margens e gate de aquisição.
- Server: APIs de benchmarks, previsões, publicações e calibrações.
- Publisher: adapters determinísticos e bundles assinados.
- Web/reports: narrativa, evidências, gargalo, intervalos, N+1 e bloqueios.

## Fluxo

1. Workload é normalizado segundo capacidades declaradas pelo Perceptrum.
2. Demanda é decomposta por estágio.
3. Observações compatíveis e calibrações aprovadas são selecionadas.
4. Cada âncora produz uma previsão por estágio.
5. O menor resultado recebe correção de erro e reserva operacional.
6. O gate verifica cobertura, qualidade, compatibilidade e redundância.
7. UI e relatórios recebem os mesmos dados estruturados.

## Pontos de risco

- Não reutilizar scores entre estágios.
- Não considerar especificação nominal como medição sustentada.
- Não permitir que catálogo/price refresh altere capacidades sem evidência.
- Não quebrar relatórios ou cenários existentes.

## Validação

- Unitários matemáticos e property tests de monotonicidade.
- Migração v5→v6 em cópia temporária.
- API e relatórios golden.
- Bundle/assinatura/anti-rollback.
- Typecheck, testes, build, pacote e smoke desktop.
