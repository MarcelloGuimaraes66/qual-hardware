# Memória - não confundir relatório e anexo

## Causa

O sistema possuía dois botões chamados apenas `PDF`. O operador acionou o segundo, que corretamente gerava o anexo neutro, mas a interface não deixava clara a diferença. O primeiro botão também salvava o relatório principal com nome incorreto.

## Solução permanente

- O relatório de recomendações é a exportação PDF primária e salva `qual-hardware-recomendacoes.pdf`.
- O anexo neutro permanece separado e recolhido, com aviso inequívoco.
- Nomes e textos vivem em `src/web/reportExports.ts` e possuem teste dedicado.
- O PDF principal continua usando `src/server/referencePdfReport.ts`, baseado no formato do commit `0846ff7`.

## Estado

99 testes, build, pacote e smoke macOS aprovados. Nenhum dado persistido foi alterado. Validação nativa Windows/Ubuntu permanece futura.
