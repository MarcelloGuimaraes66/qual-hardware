# Research — Calibration T4.1

## Problem

The current autonomous kernel is local and append-only, but progress lacks elapsed/ETA fields, phase files accumulate until terminal cleanup, cancel is not idempotent, interrupted runs cannot resume, and the renderer exports plain JSON using a gzip filename. Autonomous imports also require a source session, preventing safe multi-machine consolidation.

## Options considered

1. Replace calibration wholesale. Rejected: unnecessary regression surface and would discard already-green T4 work.
2. Add a compatibility layer around the current kernel, store, and UI. Selected: preserves working behavior while introducing v2 contracts and additive tables.
3. Upload results to a central service. Rejected: conflicts with the fixed offline/privacy requirement.

## Recommendation

Extend existing types and append-only storage; add dedicated progress, checkpoint, temporary-file lifecycle, and signed-exchange modules. Use canonical JSON plus Ed25519 and gzip bounds. Keep final evidence outside the temporary workspace. Make exact measured configuration a separate recommendation path so it does not inherit the three-distinct-system extrapolation gate.

## Platform equivalence

Use Node/Electron cross-platform primitives for paths, monotonic time, gzip, Ed25519, SQLite, and filesystem capacity. Platform-specific differences are limited to hardware probes, process termination, secure-key protection capability, and packaged runtime assets.

The candidate inventory is recorded in `resources/calibration/asset-sources.lock.json`; it is an audit lock, not an approval list and never authorizes runtime download. Immutable candidates were verified against official upstream metadata:

- MediaMTX v1.18.2 official release at commit `c05c14d9afcee5f662c81762eae68f85b36b70d7`, with distinct verified darwin-arm64, linux-amd64 and windows-amd64 archive hashes (`https://github.com/bluenviron/mediamtx/releases/tag/v1.18.2`).
- llama.cpp b9637 at commit `aedb2a5e9ca3d4064148bbb919e0ddc0c1b70ab3`, with distinct official macOS arm64, Ubuntu x64 and Windows CPU x64 archive hashes (`https://github.com/ggml-org/llama.cpp/releases/tag/b9637`).
- Qwen3-VL 2B revision `52d6c8ffea26cc873ac5ad116f8631268d7eb503` and 4B revision `1cd86afb9a95c410a6038ab3b40d8b578c892266`, using Q4_K_M language and Q8_0 vision artifacts with exact LFS SHA-256/size metadata (`https://huggingface.co/Qwen/Qwen3-VL-2B-Instruct-GGUF`, `https://huggingface.co/Qwen/Qwen3-VL-4B-Instruct-GGUF`).

The telemetry probe is now a first-party Go 0.1.0 implementation with a locked source-tree digest and reproducible macOS arm64, Windows x64 and Linux x64 builds. It uses only fixed local commands/files and never elevates privileges: macOS `pmset` plus Apple GPU inventory, Linux thermal zones/throttle counters/cooling state, Windows thermal-policy CIM data, and NVIDIA thermal slowdown reasons when present. Missing or incomplete CPU/GPU evidence is labeled partial/unavailable and fails closed. Candidate provenance notices and CycloneDX files are packaged, while legal/SBOM approval and native physical sensor validation remain explicit blockers.

The source set is about 4.55 GB per target. The conservative acquisition-plus-staging peak is about 9.64–9.65 GB. The final audit observed about 12.62 GB free against a 53,687,091,200-byte mandatory reserve, so acquisition/provisioning was correctly refused without downloading candidate assets.
