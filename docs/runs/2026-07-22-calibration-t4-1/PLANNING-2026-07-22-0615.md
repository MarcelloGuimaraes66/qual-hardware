## Goals and acceptance

- Progress v2 is monotonic, reconstructable, and reaches 100% only after durable persistence and cleanup.
- Disk reserve and per-phase reclamation prevent calibration-owned temporary growth from exhausting the system volume.
- Cancellation is idempotent and retains a durable diagnostic/checkpoint.
- Resume reuses only compatible discovery/checkpoint evidence; purchase-grade qualification always restarts repetition 1.
- `.qhcal` and `.qhcalset` are bounded, gzip-compressed, signed, portable, duplicate-safe, and importable without the source session.
- Recommendation precedence is exact trusted evidence, safe multi-system extrapolation, public planning evidence, then theoretical planning.
- The public catalog pauses during calibration and retries immediately afterward.
- No calibration accesses Perceptrum or sends calibration data externally.

## Risks and controls

- Database risk: additive schema v2, verified backup, before/after integrity checks, transaction-only installation.
- Filesystem risk: canonical direct-child UUID workspace, regular files only, manifest ownership, no recursive root deletion.
- Signature/import risk: compressed/decompressed byte ceilings, schema/digest/signature verification before transaction.
- Scientific risk: exact compatibility digest, immutable lineage, no partial commercial-run continuation.
- Platform risk: common code plus platform adapters; native physical validation remains a separate hard gate.

## Blast radius and change budget

- Allowed: calibration contracts, worker/kernel, temporary storage, local API, additive SQLite schema/store, UI, exchange, recommendation evidence policy, catalog coordination, tests, packaging, and run documentation.
- Excluded: Perceptrum, its data/runtime, unrelated report behavior, production deployment, and user data deletion.
- Change budget: targeted redesign inside the declared calibration/evidence boundaries.

## Rollback criteria

Disable the affected feature flag and preserve all additive records if migration, persistence, cleanup ownership, signature verification, recommendation safety, or platform validation regresses. Never downgrade destructively.
